PROJECT OVERVIEW
────────────────
App name   : Blood Setu (blood_setu)
Type       : Flutter mobile app (Android primary)
Purpose    : Emergency blood donation platform — connects donors with recipients
             in Bangladesh. Users can request blood, find nearby donors, chat,
             and track their donation history.
Firebase   : Project ID = emergency-blood-donation-141bd

TECH STACK
──────────
Flutter + Dart
Firebase Auth      (Google Sign-In)
Cloud Firestore    (database)
GetIt + Injectable (dependency injection, code-gen)
BLoC + Freezed     (state management, code-gen)
Dartz              (Either<Failure, T> for error handling)
GoRouter           (navigation with auth guards)
Geolocator         (GPS location)
SharedPreferences  (local persistence via SpService)
intl               (date formatting)

Run code generation:
  dart run build_runner build --delete-conflicting-outputs

FIRESTORE COLLECTIONS
──────────────────────
profile/{uid}
  Fields: user_uuid, fullName, gender, email, photoUrl, phone, bloodGroup,
          age, lastDonation (Timestamp), district, thana, fbId,
          updatedAt, isActive, donorTier, totalDonations
  NOTE: latitude/longitude are NOT stored here (moved to user_locations)
  Written: on registration via WriteBatch (profile + user_locations info fields together)
  Read:    by HomeBloc on started event (getProfile via RegistrationUserUseCase)

user_locations/{uid}
  Fields (two separate merge writes, never clobber each other):
    Info fields (written at registration):
      fullName, bloodGroup, district, thana, isActive, donorTier,
      totalDonations, photoUrl, infoUpdatedAt
    GPS fields (written on every app open):
      latitude, longitude, geohash, locationUpdatedAt
  Used for: nearby donor geohash range queries (NearbyDonorsRepositoryImpl)

donations/{uid}/records/{donationId}
  Fields: hospital, bloodGroup, date (Timestamp), status, createdAt (Timestamp)
  Written: via Firestore transaction in DonationRepositoryImpl.addDonation()
           — atomically writes record + increments profile.totalDonations
             + updates profile.lastDonation + recalculates profile.donorTier
  Read:    getDonationHistory() — ordered by date descending

users/{uid}
  Fields: uid, name, email, photoUrl, createdAt
  Written: on first Google sign-in
  Read:    not actively used

  String?   thana
  String?   fbId
  double?   longitude    ← in-memory only, NOT written to Firestore (toMap() excludes it)

  int?      daysToNextDonation     — days until eligible again (0 if past)

DONOR LOCATION MODEL — DonorLocationModel (donor_location_model.dart)
───────────────────────────────────────────────────────────────────────
Freezed class (const DonorLocationModel._() pattern for custom methods).
Represents user_locations/{uid} — the searchable donor index.
Fields: uid, fullName, bloodGroup, district, thana, isActive, donorTier,
        totalDonations, photoUrl, latitude?, longitude?, geohash?

  Map<String,dynamic> toInfoMap()   — info fields only (no coords), written at registration

AUTH FLOW
─────────
1. Splash → AuthController._init() reads SharedPreferences (profileCompleted?)
            waits for Firebase authStateChanges + 3s splash + location permission
2. Sign In → Google OAuth → Firebase Auth → creates users/{uid} if new
             checks profile/{uid} exists → returns bool (profileExists)
             AuthController.onLoginSuccess(profileExists: bool)
3. Registration (if no profile) → 2-step form
             Step 1: fullName, gender, phone, bloodGroup, age
             Step 2: district, thana, confirmation checkbox
             Optional: lastDonation (DateTime picker), fbId
             On submit: WriteBatch writes profile/{uid} + user_locations/{uid} info fields
             AuthController.onProfileCompleted()
4. Home → HomeBloc.started() → fetches profile/{uid} + fire-and-forget GPS location update
5. Logout → SignInBloc.signOut() → AuthController.logout() → GoRouter redirects


HOME SCREEN FLOW
────────────────
HomeScreen creates: getIt<HomeBloc>()..add(HomeEvent.started())
HomeBloc.started():
  - reads AuthController.user.uid
  - calls RegistrationUserUseCase.getProfile(uid) → emits HomeState with profile
  - calls unawaited(locationUseCase.updateLocation(uid)) — fire-and-forget GPS update

HomeState fields: showSidebar, sosPressed, UserProfileModel? profile

_WelcomeCard reads from profile:
  name            = profile?.fullName ?? 'Guest'
  tier            = profile?.donorTier ?? ''
  donations       = profile?.totalDonations ?? 0
  days to donate  = profile?.daysToNextDonation ?? 0
  isActive        = profile?.isActive ?? false


LOCATION UPDATE FLOW
─────────────────────
LocationRepositoryImpl.updateLocation(uid):
  1. Checks location service enabled
  2. Checks/requests permission
  3. Geolocator.getCurrentPosition(accuracy: high)
  4. Writes {latitude, longitude, geohash, locationUpdatedAt} to user_locations/{uid} (merge:true)
  Returns Either<Failure, void> — failures silently ignored by HomeBloc (best-effort).


DONORS SCREEN — REAL FIRESTORE (donors feature)
─────────────────────────────────────────────────
DonorsScreen (bottom nav tab 1) — Find Donors with real data.

DonorsBloc (@injectable) depends on NearbyDonorsUseCase.
  Events:
    started / refreshed          — load from Firestore (radius ring 0 = 5km)
    loadMoreRequested            — widen radius to next ring (5→10→25→50→100km)
    searchChanged(String)        — local filter only (instant, no Firestore)
    bloodGroupSelected(String)   — local filter only (instant, no Firestore)
    distanceSelected(String)     — re-fetch with new radius from Firestore
    filtersOpened / filtersClosed
    filtersApplied               — close sheet + apply local filters (Apply button)
    filtersReset                 — reset all + Firestore re-fetch

  State fields: status, donors (all loaded), filtered (locally filtered),
                hasReachedMax, radiusIndex, search, selectedBloodGroup,
                selectedDistance, showFilters, errorMessage

  Filter design:
    Blood group → LOCAL filter on loaded donors (no Firestore composite index needed)
    Search text → LOCAL filter on loaded donors
    Distance    → changes max radius → Firestore re-fetch
    "Apply" btn → applies local filters (blood group + search) + closes sheet

NearbyDonorsRepositoryImpl.getNearbyDonors():
  - Computes geohash range bounds for the radius circle (GeoQueryUtil.queryBounds)
  - Runs one Firestore query per range concurrently (orderBy geohash + startAt/endAt)
  - Merges, dedupes by uid, drops docs outside true circle, sorts by distance
  - NO bloodGroup in Firestore query — filtered locally after fetch

NearbyDonorsRepositoryImpl.getOrigin(uid):
  - Reads user_locations/{uid} for the current user's lat/lng (set by LocationUseCase)
  - Returns failure if lat/lng null (GPS not yet written)

DonorCard in donors_screen.dart uses NearbyDonor fields:
  name, bloodGroup, thana+district, distanceKm, totalDonations, isActive
  Avatar: photoUrl if available, else Avatar(initials, colorHex derived from uid.hashCode)
  Status chip: isActive → 'Available' (green) / 'Unavailable' (red-surface)


DONATION RECORD FLOW
──────────────────────
DonationRepositoryImpl.addDonation(uid, entry):
  Firestore transaction:
    1. Read current profile.totalDonations
    2. newCount = currentCount + 1
    3. Write entry.toMap() to donations/{uid}/records/{auto-id}
    4. Update profile: totalDonations = newCount, lastDonation = entry.date,
       donorTier = _computeTier(newCount)

donorTier thresholds (computed in DonationRepositoryImpl._computeTier):
  '' (empty) = 0 donations
  Bronze     = 1–5
  Silver     = 6–11
  Gold       = 12–24
  Platinum   = 25+


PROFILE SCREEN — LOGOUT FLOW
─────────────────────────────
ProfileScreen provides: ProfileBloc + SignInBloc (from getIt)

Logout tap:
  1. Utils.showConfirmDialog → "Are you sure?" (red Logout btn + Cancel)
  2. If confirmed → SignInBloc.add(SignInEvent.signOut())
  3. BlocConsumer<SignInBloc> listener:
     - loading   → SparkleLoadingOverlay shown in Stack
     - success   → Utils.showSuccessDialog (auto-navigated away by AuthController)
     - failure   → Utils.showErrorDialog

IMPORTANT BUG FIX: dartz Either.fold() is synchronous. If callbacks are async,
  you must `await result.fold(...)` to prevent "emit after handler completed" crash.
  Both callbacks must be `async` for this to work.


REGISTRATION BLOC — IMPORTANT NOTES
──────────────────────────────────────
lastDonation is stored as DateTime? in RegistrationState (NOT String).
  - Date picker passes DateTime directly to event
  - Display formatting (dd/MM/yyyy) done in the UI only
  - Submitted to UserProfileModel as DateTime? directly
  - Reason: DateTime.tryParse() only accepts ISO 8601, not dd/MM/yyyy

result.fold() in registrationSubmitted is NOT awaited (callbacks are sync).
This is fine since both fold callbacks are synchronous.

KEY DESIGN DECISIONS
─────────────────────
1.  UserProfileModel is a plain Dart class (not Freezed) — has fromFirestore() + toMap()
2.  All other models (Donor, BloodRequest, etc.) use Freezed
3.  DonationHistoryEntry uses Freezed + custom methods via const DonationHistoryEntry._()
4.  Either<Failure, T> from dartz for all repository return types
5.  Await result.fold() when fold callbacks are async (BLoC emit safety)
6.  lastDonation stored as DateTime? in state — never as String
7.  HomeBloc uses RegistrationUserUseCase (not AuthenticationUseCase) for getProfile
8.  SparkleLoadingOverlay is a Positioned.fill widget — must be inside a Stack
9.  GPS is NOT fetched at registration — only on app open via LocationRepositoryImpl
10. Location updates are fire-and-forget — failures silently ignored (best-effort)
11. totalDonations is updated via Firestore transaction in addDonation — never manually set
12. donorTier is stored as plain String: '', 'Bronze', 'Silver', 'Gold', 'Platinum'
13. GeohashUtil is pure Dart (no external package) — used for user_locations geo-queries
14. Blood group + search filtering is LOCAL (on loaded donors) — no Firestore composite index needed
15. Distance filter changes the Firestore query radius — triggers re-fetch
16. All BLoC event.when() handlers must be async when using await event.when() — mixed
    sync/async handlers make when() return void, breaking the await
17. user_locations has TWO separate merge writes: info (registration) + coords (app open)
    They never clobber each other because both use SetOptions(merge:true)
